# Senior Workflow

Use estas regras sempre que Codex atuar neste projeto. O objetivo e manter o bot seguro, organizado, simples de evoluir e economico em tokens.

## Regra Mestre

Use o Senior Workflow do projeto. Aplique Clean Architecture, Defensive Programming, Security First, KISS, Test Before Trust, Bot Safety e Database Safety. Antes de editar, leia o contexto com `rg`. Faca mudancas pequenas, preserve o estilo do projeto, nao vaze secrets, valide entradas e rode testes.

## Skills Fixas

- Senior Workflow: entender antes de editar, escolher o menor caminho seguro, testar e explicar.
- Clean Architecture: manter dominio, aplicacao, infraestrutura, UI e config separados.
- Defensive Programming: validar toda entrada externa e falhar com erro claro.
- Security First: proteger tokens, senhas, certificados, URLs privadas e permissoes.
- KISS: evitar complexidade, dependencia ou abstracao sem necessidade real.
- Test Before Trust: confiar em teste e verificacao, nao em achismo.
- Bot Safety: proteger fluxos de Discord contra loop, permissao errada e interacao expirada.
- Database Safety: preservar historico e fazer alteracao manual com `SELECT` antes de `UPDATE`/`DELETE`.

## Principios

- Clean Architecture: manter regras de dominio separadas de Discord, banco, CLI e infraestrutura.
- Defensive Programming: validar entradas, tratar falhas previsiveis e evitar dados invalidos no banco.
- Security First: proteger credenciais, permissoes administrativas e dados sensiveis.
- KISS: resolver pelo caminho mais simples que preserve clareza, seguranca e manutencao.
- Test Before Trust: todo bug corrigido deve ter teste ou verificacao objetiva.
- Bot Safety: interacoes Discord devem ser idempotentes, autorizadas e amigaveis em erro.
- Database Safety: preservar historico e alterar dados com filtro claro.
- Menor mudanca util: alterar apenas o necessario para o pedido atual.
- Evidencia antes de edicao: ler codigo, testes e fluxo existente antes de propor ou aplicar mudancas.

## Fluxo Obrigatorio

1. Entender o pedido e identificar o comportamento esperado.
2. Procurar contexto com `rg` antes de abrir muitos arquivos.
3. Ler os arquivos diretamente relacionados ao fluxo.
4. Localizar testes existentes ou criar teste focado quando houver risco.
5. Implementar pequeno, no padrao do projeto.
6. Rodar verificacao relevante, de preferencia `npm test` ou teste focado.
7. Responder com o que mudou, onde mudou e como foi validado.

## Arquitetura do Bot

- `src/index.js` deve continuar como composition root e orquestrador de eventos.
- Regras de negocio devem ficar em `src/modules/*` ou `src/domain/*`.
- Persistencia deve ficar concentrada em `src/shared/database.js`.
- UI do Discord deve ficar em `src/core/discordUi.js`.
- Configuracao deve passar por `src/core/config.js`, evitando `process.env` espalhado.
- Parsers devem ser puros sempre que possivel: entrada texto, saida objeto, sem efeitos colaterais.

## Clean Architecture

- Nao deixar modulos de dominio dependerem diretamente de Discord.js.
- Passar dependencias por parametro quando isso facilitar teste.
- Separar parsing, validacao, persistencia e apresentacao.
- Evitar misturar regras de ranking, membros, tickets, formulario e disciplina no mesmo bloco.
- Criar abstracao apenas quando houver duplicacao real ou acoplamento dificil de manter.

## Defensive Programming

- Nunca confiar em entrada de usuario, modal, embed, mensagem ou variavel de ambiente.
- Validar IDs antes de salvar: `playerId` deve ser numerico quando representar ID in-game.
- Normalizar texto antes de comparar labels, nomes de canais ou comandos.
- Tratar valores ausentes com erro claro ou fallback explicito.
- Evitar gravar payload parcial quando faltar campo obrigatorio.
- Nao vazar `.env`, tokens, senha de banco, certificado, URL privada ou segredo em logs/respostas.
- Em SQL, usar parametros, nunca interpolar valor de usuario em query.
- Em migracoes/DDL automatico, usar `IF NOT EXISTS` e alteracoes idempotentes.

## Security First

- Nunca imprimir ou repetir conteudo de `.env`, token Discord, senha Postgres, certificado ou URL privada.
- Nunca commitar arquivo com segredo.
- Validar permissao antes de acao administrativa, revisao, ticket, cargo, nickname ou fechamento.
- Preferir mensagens de erro genericas ao usuario e logs tecnicos sem segredo.
- Tratar anexos, embeds, webhooks e mensagens de bot como entrada nao confiavel.
- Em comandos destrutivos, exigir filtro claro e explicar risco antes.

## KISS

- Preferir uma funcao clara a uma cadeia de helpers genericos.
- Preferir regex pequena e testada a parser complexo quando o formato for simples.
- Preferir teste unitario focado a mock gigante.
- Evitar dependencia nova se Node.js ou o codigo existente ja resolve bem.
- Nao otimizar performance sem evidencia.
- Nao refatorar arquivo inteiro para corrigir uma linha.

## Economia de Tokens

- Antes de pedir contexto ao usuario, buscar no repo.
- Carregar somente os arquivos relacionados ao problema.
- Resumir arquivos longos em vez de colar grandes trechos.
- Fazer diffs pequenos e explicaveis.
- Evitar respostas longas quando uma query, comando ou patch pequeno resolve.
- Reusar conhecimento do projeto ja descoberto na conversa.

## Banco de Dados

- Preservar historico de `bvb_actions`; nao sugerir `TRUNCATE` nessa tabela sem aviso forte.
- Manter tabelas operacionais separadas: `members`, `tickets`, `ticket_events`, `application_forms`, `transfer_requests`, `disciplinary_actions`, `point_sessions`, `point_events`.
- Validar dados antes do insert/update e tambem perto da origem quando possivel.
- Ao corrigir dados de producao, preferir `SELECT` de auditoria antes de `UPDATE` ou `DELETE`.
- Para updates manuais, usar `WHERE id = ...` ou chave unica clara.

## Testes

- Para parser: testar formato valido, formato alternativo e entrada invalida.
- Para persistencia: testar validacao e parametros enviados ao banco.
- Para UI Discord: testar payload essencial, nao detalhe visual irrelevante.
- Para bug corrigido: adicionar teste que falharia antes da correcao.
- Rodar teste focado primeiro; rodar suite completa quando mexer em fluxo compartilhado.

## Bot Safety

- Nao responder ao proprio bot quando isso puder criar loop.
- Permitir webhooks/mirrors apenas nos canais configurados para logs.
- Checar permissao antes de aprovar/reprovar, assumir ticket, fechar ticket, gerenciar acesso ou aplicar cargo.
- Usar resposta ephemera para erro operacional ou permissao negada.
- Tratar interacao expirada sem derrubar o processo.
- Registrar evento importante em `ticket_events`, `bvb_actions` ou tabela operacional correspondente.

## Database Safety

- Nao rodar `DELETE`, `UPDATE`, `TRUNCATE` ou alteracao manual sem `SELECT` previo.
- Nunca rodar `TRUNCATE bvb_actions` sem confirmacao explicita e backup.
- Usar `WHERE id = ...`, chave unica ou escopo `instance_id + guild_id`.
- Preferir correcao pontual a limpeza ampla.
- Preservar rastreabilidade com `source_channel_id`, `source_message_id`, `created_at` e payloads.

## Padrao de Resposta

- Dizer primeiro o resultado pratico.
- Citar arquivos alterados com caminho.
- Informar verificacao rodada.
- Avisar riscos restantes ou dados antigos que precisam correcao manual.
- Usar exemplos curtos de SQL/comando quando o usuario precisar agir no Beekeeper, terminal ou Discord.
